# Skill 软连接扫描设计

## 背景

`SkillManager` 使用 `readdir` 的 `Dirent` 结果扫描全局 Skill 目录。目前只识别普通目录和普通文件。Node.js 对软连接返回 `isSymbolicLink() === true`，因此指向目录的 Skill 和指向 Markdown 文件的 Skill 都会被漏掉。

## 目标

扫描以下软连接形式，并保持现有 Skill 摘要路径语义：

- 根 Skill 目录下指向目录的软连接；
- 根 Skill 目录下指向 `.md` 文件的软连接（仅对启用根 Markdown 的位置）；
- Skill 目录递归扫描过程中遇到的目录软连接；
- 软连接目录形成循环时不无限递归。

## 设计

扫描目录项时，普通项沿用 `Dirent` 的类型判断；软连接通过跟随目标的 `stat` 解析为文件或目录。目标不存在、权限不足或其他 `stat` 错误的软连接跳过，不让单个坏链接阻断整个位置的扫描。

递归进入目录前，通过 `realpath` 得到目标的 canonical 路径，并维护当前递归链上的路径集合。若目标已经在当前递归链中，则跳过该分支；递归返回后移除路径。这样可阻止 `a -> b -> a` 等循环，同时不会把来自不同入口的同一个 Skill 错误去重。

读取 Skill 时继续使用用户实际配置的路径，而不是 canonical 路径。这样 UI 展示、编辑、重命名和删除行为保持与现有路径一致；文件系统 API 本身会自然跟随软连接。

## 错误处理

- 目录读取失败：沿用当前行为，跳过该目录。
- 软连接目标 `stat` 失败：跳过该项。
- `realpath` 失败：跳过该递归分支。
- 软连接目标类型不符合 Skill 扫描规则：忽略。

## 测试

使用项目现有 Node 原生测试约定添加回归测试，创建临时目录和软连接，覆盖：

1. 目录软连接被发现并读取 `SKILL.md`；
2. 根目录 Markdown 文件软连接被发现；
3. 递归目录中的目录软连接被发现；
4. 循环软连接不会导致扫描无限递归；
5. 不支持创建软连接的环境明确跳过测试。

验证命令包括针对性回归测试和 `npm run typecheck`。
