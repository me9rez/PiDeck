01 目标与范围

核心目标

  1. 跨 Agent 状态感知：用户在任意软件全屏工作时，余光就能看到当前所有 Agent 的聚合状态（空闲/工作/出错/完成），无需切回 Pideck 主窗口。
  2. 零成本美观：复用 petdex 的 pet.json + spritesheet 素材格式与生态，开箱自带可切换的宠物，渲染层自研（Canvas 精灵图切帧）。
  3. 三端一致体验：macOS、Windows、Linux 三个平台都能透明悬浮、可拖拽、可点击跳回。
  4. 叠加式、可关闭：默认开关，关闭后应用与现状完全一致，不触碰三栏主界面与现有 IPC。

范围划分

✅ 本期做

核心功能

全局聚合状态机 · 透明悬浮窗（三端）· Canvas 渲染 · 内置宠物 + petdex 兼容 · 设置面板 · 拖拽/点击交互 · 跳转活跃 Agent

⏭ 后期可选

增强项（不影响 MVP）

逗弄/摸头动画 · 自动巡游桌面 · 开机自启 · 宠物市场（一键下载社区包）· 通知气泡（出错/完成时宠物头顶提示）

🎯
为什么比 Codex 单 Agent 桌宠更有价值

Codex 只跑一个 Agent，宠物纯装饰；Pideck 管多个 Agent，「不用切窗口就知道哪个在跑、哪个出错」是真实痛点。宠物相当于一个跨 Agent 的灵动岛。

03 全局聚合状态机 ★ 决策 C 核心

这是本设计最关键的部分——多 Agent 如何聚合成一只宠物的一个状态。

3.1 聚合规则（优先级从高到低）

遍历所有 Agent 的 AgentStatus（types.ts:51：starting/idle/running/error/closed），按下表优先级取首个命中作为全局状态：

序触发条件聚合状态含义
1任一 Agent.status === errorfailed有 Agent 出错，最该提醒
2任一 Agent.status === runningrunning有 Agent 正在工作
3任一 Agent.status === startingwaiting有 Agent 启动中
4所有非 closed 的 Agent 都是 idleidle全部就绪空闲
5全部 closed 或无 Agenthidden没有活跃 Agent，宠物隐藏
⚠️
为什么 error 优先级最高

出错是需要用户立即介入的信号，不能被「同时有别的 Agent 在 running」淹没。即便有 3 个 Agent 在跑、1 个崩了，宠物也应显示 failed，提示用户去看。

3.2 AgentStatus → 宠物动画行映射

宠物 spritesheet 沿用 petdex/Codex 的 9 行状态约定（每行 8 帧）。Pideck 的 5 个 AgentStatus 映射如下（聚合状态与映射共用同一张表）：

← AgentStatus: running（聚合主态）

🥺waiting行6

← AgentStatus: starting

➡️running-right行1

📌

Pideck 当前 AgentStatus 只有 5 态，没有「等待审批」（pi 的审批走 tool 消息，不进状态机，见 AgentManager.ts:920 tool_execution_*）。MVP 不做 review 态；若后期想细化，可订阅 agents:event（ipc.ts:85）按 event.type 推断，预留了行8。

3.3 去抖与状态跳变防护

问题

多 Agent 状态高频抖动

多个 Agent 同时启停时，聚合状态可能在 running↔idle 间快速跳动，宠物动画会抽搐。

对策

150ms 去抖 + 动画完成锁

状态变化后等 150ms 无新变化才推送；进入新状态后至少播放完一个动画周期（~600ms）才允许切下一个，避免半帧切换。

// PetStateBridge 聚合核心（伪代码，见第 10 节完整契约）
const PRIORITY = ["error", "running", "starting", "idle"];  // closed→hidden

function aggregate(agents: AgentTab[]): PetAggregateState {
  const active = agents.filter(a => a.status !== "closed");
  if (active.length === 0) return { mode: "hidden" };

  const has = (s: AgentStatus) => active.some(a => a.status === s);
  let mode: PetMode =
      has("error")    ? "failed"   :
      has("running")  ? "running"  :
      has("starting") ? "waiting"  : "idle";

  return {
    mode,                                          // 当前聚合状态
    runningCount: active.filter(a=>a.status==="running").length,
    errorCount:   active.filter(a=>a.status==="error").length,
    activeAgentId: pickFocusAgent(active),          // 点击宠物跳转目标
  };
}

focus agent 选取（点击宠物跳到谁）

聚合状态下，点击宠物应跳到「最值得关注」的那个 Agent：

  • 有 error → 跳第一个出错的；
  • 否则有 running → 跳最近开始 running 的；
  • 否则跳最近活跃（createdAt 最大）的 Agent。

04 数据源与桥接

4.1 关键约束：emit 是单窗口的

🚨
最重要的实现约束

AgentManager.emit()（AgentManager.ts:1519）只向 getWindow() 返回的 mainWindow 发送（index.ts:1637 注入 ()=>mainWindow）。独立的宠物 BrowserWindow 收不到任何 agent 事件。

解决方案：宠物窗口不直接订阅 agent 事件，而是由主进程的 PetStateBridge 在主窗口侧订阅 agents:state，聚合后通过新的 pet:state channel 主动推送给宠物窗口的 webContents。

5.2 三端差异处理矩阵

平台透明窗支持关键配置已知坑 / 降级
macOS 原生完美 transparent:true 即可；visibleOnAllWorkspaces:true 跨桌面空间；置顶用 "floating"（index.ts:1232 已验证） 无。参考点最多、体验最佳。
Windows 支持，需处理黑边 transparent:true + backgroundColor:"#00000000"；Win11 可用 backgroundMaterial:"acrylic"（宠物本身不需要材质，留空）；alwaysOnTop 正常 ① 老显卡/远程桌面下透明可能渲染黑块 → 降级：检测失败则改用纯色背景 + sprite 带圆角 mask；② 点击穿透需 setIgnoreMouseEvents（见 5.3）
Linux 视 WM 而定 transparent:true + frame:false；X11 下置顶 "floating" 多数 WM 支持；Wayland 需 gtk-titlebar 模式 ① Wayland 部分合成器不支持透明无边框 → 降级：渲染带 1px 边框的圆角窗，或退化为托盘图标状态点；② 自由定位受限（Wayland 无全局坐标）→ 启动时居中，拖拽存相对偏移
🧪
三端验证策略

实现时按 mac → Windows → Linux 顺序适配。每端写一个能力探测：detectPetWindowCaps() 返回 {transparent, clickThrough, freePosition} 三个布尔，渲染层据此选择「透明悬浮」/「圆角小窗」/「托盘点」三种渲染形态，保证三端都能显示状态，只是精致度递减。

5.3 点击穿透（click-through）

透明区域应允许鼠标穿透到下层应用，宠物本体可点击。

  • 实现：渲染层用 requestAnimationFrame 检测鼠标是否在 sprite 像素上（hit-test），调用 api.pet.setIgnoreMouseEvents(true, {forward:true}) 切换穿透。
  • 简化方案（MVP）：不穿透，整个小窗可点击/拖拽（窗很小，160×176，对桌面占用极小）。三端行为一致，避免 hit-test 的复杂度。
  • 后期：再做精确像素穿透。

5.4 生命周期与位置持久化

  • 创建时机：app ready 且 settings.petEnabled 为 true 时创建；开关关闭则销毁。
  • 位置持久化：拖拽后 petWindow.getBounds() 存入 userData/pet-position.json，下次启动恢复；超出屏幕边界则回退到右下角默认位。
  • 多显示器：拖到别的屏，存 {displayId, x, y}。
  • 主窗口关闭：主窗口隐藏到托盘（index.ts:578 closeToTray）时，宠物窗保留（这正是宠物存在的意义——主窗关了也能看状态）。

06 渲染层

6.1 PetOverlay.tsx —— Canvas 精灵图渲染

宠物窗口加载一个极简 HTML（pet.html），只渲染一个 Canvas + PetOverlay 组件。

// 渲染核心：按 mode 选行，循环切帧
const ROWS: Record<PetMode, number> = {
  idle: 0, running: 7, failed: 5, waiting: 6, waving: 3,
};
const FRAMES_PER_ROW = 8, FPS = 10;

function PetOverlay({ sprite, mode }: Props) {
  const row = ROWS[mode];
  const frame = useFrameLoop(FPS);          // 0..7 循环
  const sx = (frame % FRAMES_PER_ROW) * FRAME_W;
  const sy = row * FRAME_H;
  ctx.clearRect(0,0,W,H);
  ctx.drawImage(sprite, sx, sy, FRAME_W, FRAME_H, 0, 0, W, H);
}
帧率

10 fps

idle 时降到 6fps 省电；running/failed 维持 10fps。Codex 实测 8-12fps 足够流畅。

尺寸

160×176 px

单格 192×208 缩放到窗口尺寸，DPR 适配高分屏。

背景

纯透明

Canvas 清屏后只画 sprite，透明区域露出窗口透明背景。

6.3 PetInteraction.tsx —— 拖拽与点击

  • 拖拽：mousedown → 记录偏移 → mousemove 调 api.pet.moveWindow(x,y)；mouseup 存位置。
  • 点击（非拖拽）：mouseup 时位移 < 3px 视为点击 → api.pet.focusAgent() → 主窗口 show()+focus() 并切到 focus agent tab。
  • 逗弄（增强项）：双击触发 jumping 动画一次。

CSP 处理（项目当前无 CSP）

🔒
现状：项目完全未配置 CSP

仓库内 grep Content-Security-Policy / onHeadersReceived 零命中（renderer/index.html 无 meta，主进程无 session 头）。现有 app 靠 loadFile（index.ts:641）加载本地资源。

宠物窗加载 file:// sprite 的两种方式：

① 内置包：走 ?asset 导入或 Vite 打包，渲染层用相对 URL，不受限；
② petdex 磁盘包：用 file:// 绝对路径加载 → 需在宠物窗 session 显式加 CSP：img-src 'self' file: data:。这是项目第一个 CSP 配置，仅作用于宠物窗 session，不影响主窗。

// PetWindow.ts：仅给宠物窗 session 加 CSP，主窗不动
petWindow.webContents.session.webRequest.onHeadersReceived((details, cb) => {
  cb({
    responseHeaders: {
      ...details.responseHeaders,
      "Content-Security-Policy": [
        "default-src 'self'; img-src 'self' file: data:; script-src 'self'",
      ],
    },
  });
});
