01 先纠正一个关键认知

你说「使用 petdex.dev 中已有的开源动画库」。调研后发现这里有一个需要点明的认知偏差，它会直接影响技术选型：

▸ 你的预期

petdex 是一个动画渲染库

像 framer-motion / lottie 那样，npm 安装后引入即用，负责把动画画出来。

▸ 实际情况

petdex 是素材仓库 + 格式规范 + 安装器

它提供标准素材包和下载能力，不含渲染代码。渲染需要我们自己在 Pideck 里写。

petdex 真正提供的三样东西

  1. 标准素材格式：一个 pet.json + 一张 spritesheet.webp/png
  2. 3000+ 现成宠物包：npx petdex install boba 一条命令下载
  3. 一个桌面 sidecar：独立进程监听 127.0.0.1:7777/state，POST 状态 JSON 驱动宠物
💡
这意味着什么

petdex 的素材格式和生态可以直接复用，但渲染层（把 spritesheet 画成会动的动画）需要 Pideck 自己实现。好消息是——这非常简单，Canvas 或 CSS 切帧几十行代码搞定，8–12fps 足够流畅。

02 Codex 桌宠的本质

核心一句话：状态机驱动的精灵图（Sprite Sheet）悬浮窗。

三大特征

  • 全局悬浮置顶：独立透明窗口浮在所有 App 之上，切到任何软件都能看见
  • 9 种状态严格映射 Agent 状态（不是随机播放，是状态机绑定）
  • 可交互：拖动改位置、点击跳回主界面、鼠标逗弄触发跳跃

素材格式规范（Codex Pet Contract）

# 精灵图集 Spritesheet 规格
整张尺寸    1536 × 1872 像素
网格        8 列 × 9 行
每格尺寸    192 × 208 像素
背景        透明（未使用的格子必须全透明）

# 一个宠物包 = 两个文件
~/.codex/pets/boba/
  ├─ pet.json           # 元数据：id / displayName / description
  └─ spritesheet.webp   # 精灵图集
// pet.json 极简结构
{
  "id": "boba",
  "displayName": "Boba",
  "description": "拿着珍珠奶茶的小水獭",
  "spritesheetPath": "spritesheet.webp"
}

03 Pideck 可行性：非常高

关键原因——Pideck 的底子几乎是量身定做的。需要的能力大部分已经具备：

需求项Pideck 现状差距
透明置顶悬浮窗Electron 原生支持 transparent + alwaysOnTop + frame:false新建独立 PetWindow
Agent 状态数据源✅ 已有完整状态机
starting / idle / running / error / closed		零成本，只需订阅
状态推送机制✅ AgentManager 已 emit
agentsState / agentsRuntimeState		直接复用
配置 / 多语言 / UI 规范✅ SettingsStore / i18n / 设计规范齐全加一个 Pets 设置项
宠物素材生态❌ 暂无复用 petdex 格式 + 内置几只
🎯
比 Codex 更有价值的点

Codex 是单 agent，宠物只是装饰。而 Pideck 管理多个 agent，宠物能解决真实痛点——「不用切窗口就知道哪个 agent 在跑、哪个出错了」，相当于一个跨 agent 的灵动岛。

04 状态映射方案

Pideck 的 AgentStatus 直接映射到宠物状态：

starting→waiting（启动中，等待就绪）
idle→idle（空闲发呆）
running→running（工作中）
error→failed（出错了）
closed→waving / 隐藏（收工）
⚠️
多 agent 聚合策略（待定）

Pideck 同时管多个 agent，但宠物只有一个。需要明确聚合规则，避免状态混乱跳动。候选：「任一 running → running；任一 error → failed；全 idle → idle」，或只跟随当前激活的 agent。这个需要你拍板（见第 9 节）。

05 技术方案概览

新增 5 个模块，全部是叠加式改动，不破坏现有三栏主界面：

src/main/pet/ ├─ PetWindow.ts # 透明置顶悬浮窗（多平台差异处理） [新] ├─ PetManager.ts # 扫描并加载 pet 包（内置 + ~/.codex/pets/） [新] └─ PetStateBridge.ts # 订阅 AgentManager 状态 → 映射 → 推送 [新] src/renderer/pet/ ├─ PetOverlay.tsx # Canvas 精灵图渲染 + 状态机 [新] └─ PetInteraction.tsx # 拖拽 / 点击 / 逗弄交互 [新] settings: 外观 → 宠物 tab [新]

06 风险与应对

风险说明应对
多平台透明窗macOS 支持最好；Windows 需处理点击穿透 + 黑边；Linux 各窗口管理器差异大分平台代码路径，优先 macOS
多 agent 状态聚合Pideck 管多个 agent，宠物只有一个，状态可能频繁跳变聚合策略 + 点击宠物跳转活跃 agent
性能持续动画可能占用 CPU帧率控制 8–12fps，idle 时降频
打包内置素材需进 asarextraResources 或 assets 打包
CSP渲染层加载本地图片可能被安全策略拦截用自定义协议或 webp 内联
🛡️
没有颠覆性改动

所有改动都是新增独立模块 + 加一个设置项，主进程窗口逻辑、三栏布局、现有 IPC 全部不动。最坏情况：关掉宠物开关，应用和现在完全一样。

08 我的建议

  1. 方案可行，强烈推荐做。Pideck 作为多 agent 管理器，宠物正好能解决「不用切窗口就知道哪个 agent 在跑」的痛点，比 Codex 单 agent 场景更有价值。
  2. 渲染层自研 + 素材复用 petdex 格式，这是唯一正确的路（petdex 没有现成渲染库，只有素材规范）。
  3. 内置 + 兼容双轨：开箱即用，又能白嫖生态。用户装了 petdex CLI 后，在 Pideck 设置里就能直接选到社区宠物。
  4. 建议从 MVP 切入：先做透明窗 + 内置一只宠物 + 三态，跑通整条链路（状态桥接 → 渲染），再逐步补全 9 态和交互。
