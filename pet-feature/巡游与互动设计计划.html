决策一 · Review 语义

完成时显示 review

running → idle 转换的庆祝动画由 jumping 改为 review（替代）。jumping 动作保留给「逗弄」使用。

决策二 · 逗弄手势

双击宠物触发跳跃

双击（两次 pointerup 间隔 < 300ms）触发 jumping；保留单击 = 跳转活跃 Agent的现有行为，互不冲突。

决策三 · 巡游激活

idle 时自动巡游

无任务运行时自动沿屏幕底部左右走动；running / failed / review / 逗弄时自动让位。设置里提供开关，默认开启。

💡
为什么不监听 pi 审批事件来触发 review？

pi 的审批走 tool_execution_* 工具消息流，不在 AgentStatus 状态机里（见 AgentManager.ts:940）。本期 review 统一对应「任务完成」语义，不解析审批事件，避免引入与现有 5 态 AgentStatus 冲突的副作用。审批触发留作后续增强项。

3功能一：巡游 Patrol

3.1 优先级让位规则

巡游是低优先级状态，任何「真实业务态」出现都要立即让位。优先级从高到低：

P0failed / hidden / waving— 业务态，巡游停止
P1running / waiting— 业务态，巡游停止
P2review（完成庆祝 4s）— 临时态，巡游停止，结束后回 idle 再自动巡游
P3jumping（逗弄 2.5s）— 临时态，巡游停止
P4patrol（running-right/left）— 仅在 idle 且开关开时启动
P5idle— 巡游的触发条件本身

3.2 新增模块 PetPatrol

新建 src/main/pet/PetPatrol.ts。它持有巡游定时器，按方向平移窗口（复用 PetWindow.moveTo），并在每个 tick 推送 running-right/running-left 给宠物窗。

// src/main/pet/PetPatrol.ts （新文件 · 伪代码示意结构，落地时补类型与边界守护）
import type { BrowserWindow } from "electron";
import { screen } from "electron";
import { ipcChannels } from "../../shared/ipc";

export class PetPatrol {
  private timer: NodeJS.Timeout | null = null;
  private direction: "left" | "right" = "right";
  private readonly speedPxPerSec = 40;   // 慢速，约 2.5s 走 100px
  private readonly tickMs = 50;          // 20fps 位置更新
  private readonly edgeMargin = 16;

  constructor(private getWin: () => BrowserWindow | null) {}

  start() {
    if (this.timer) return;                // 已在巡游，幂等
    this.pushMode();                          // 先推一次方向 mode
    this.timer = setInterval(() => this.tick(), this.tickMs);
  }

  stop() {
    if (this.timer) { clearTimeout(this.timer); this.timer = null; }
    // 不强制复位窗口：停在原地，下次 idle 巡游从当前位置继续
  }

  private tick() {
    const win = this.getWin(); if (!win || win.isDestroyed()) return;
    const [x, y] = win.getPosition();
    const [w, h] = win.getSize();
    const wa = screen.getDisplayMatching({x, y, width:w, height:h}).workArea;
    const step = this.speedPxPerSec * (this.tickMs / 1000);
    let nx = this.direction === "right" ? x + step : x - step;

    // 边界翻转：撞到边缘则反向，并切换 spritesheet 行
    if (nx >= wa.x + wa.width - w - this.edgeMargin) {
      nx = wa.x + wa.width - w - this.edgeMargin;
      this.direction = "left"; this.pushMode();
    } else if (nx <= wa.x + this.edgeMargin) {
      nx = wa.x + this.edgeMargin;
      this.direction = "right"; this.pushMode();
    }
    win.setPosition(Math.round(nx), y);
  }

  private pushMode() {
    const win = this.getWin();
    if (win && !win.isDestroyed()) {
      win.webContents.send(ipcChannels.petState, { mode: this.direction === "right" ? "running-right" : "running-left" });
    }
  }
}
🧩
为什么 PetPatrol 直接 send pet:state，而不走 PetStateBridge？

巡游是纯 UI 层自主行为，与 Agent 状态无关。PetStateBridge 聚合的是业务态，让 PetPatrol 通过 send 直接驱动方向帧，可避免在 Bridge 里混入「方向」这一非业务概念。Bridge 只负责在业务态变化时调 patrol.stop()；当回到 idle，Bridge 再调 patrol.start()，由 PetPatrol 自己决定方向帧。Bridge 是巡游的「开关」，PetPatrol 是巡游的「引擎」。

3.3 边界处理与集成点

边界守护

复用 resize 的回钳思路

每个 tick 用 screen.getDisplayMatching 取当前屏 workArea，与 PetWindow.ts:185-192 的回钳一致，保证多屏/缩放后不越界、不掉到任务栏下。

集成点

PetStateBridge.push 改两处

① 目标为 idle 且 petPatrolEnabled → patrol.start()；② 目标为 running/failed/waiting/review/jumping → patrol.stop()。hidden/waving 也 stop。

设置开关

Settings（types.ts:315-323 已有的 pet 区块）新增一项：

// src/shared/types.ts · Settings 内 pet 区块

设置面板加一个开关（与现有 petAlwaysOnTop 同款 Toggle），preload 经 settings.get/set 暴露读写。开关变化时若当前正 idle 则即时启停巡游。

5功能三：逗弄 Tease

5.1 双击手势检测

改造 PetInteraction.tsx。现有 onPointerUp 在位移 < CLICK_THRESHOLD(3px) 时判定为单击。在此基础上加双击判定：两次 pointerup 间隔 < 300ms 且都属点击 → 触发逗弄；否则维持单击行为。

// PetInteraction.tsx · 在 onPointerUp 内补充双击判定
const lastTapAt = useRef(0);

const onPointerUp = (e) => {
  // ... 原有 release / onDragStateChange(false) ...
  if (moved.current < CLICK_THRESHOLD) {
    const now = Date.now();
    if (now - lastTapAt.current < 300) {
      // 双击 → 逗弄，吞掉本次单击跳转
      lastTapAt.current = 0;
      void window.piDesktop.pet.tease();
      return;
    }
    lastTapAt.current = now;
    // 延迟单击跳转，给第二次点击留 300ms 判定窗口
    const pendingClick = setTimeout(() => {
      window.piDesktop.pet.focusAgent();
    }, 300);
    pendingClickRef.current = pendingClick;
  }
};
⚠️
双击会引入 ~300ms 的单击延迟。

为判定双击，单击跳转需延迟 300ms 执行；若期间来了第二次点击则取消跳转、改为逗弄。这是「单/双击共存」的标准代价（macOS Finder 同款）。若实测觉得单击变钝，可改用「单击立即跳转 + 双击额外逗弄」的并发策略，但会出现双击时先跳转 Agent 再跳跃的副作用——建议先用延迟方案，实测后决定。

5.2 逗弄的临时状态通道

状态机在主进程，渲染层不能直接改 mode。新增一条 IPC 把「逗弄」请求送到主进程，由 PetStateBridge.tease() 注入 jumping：

双击 → pet.tease() · preload → pet:tease · IPC → PetStateBridge.tease() → jumping 2.5s → 恢复真实态 
// PetStateBridge.tease() · 逗弄注入 jumping，结束后恢复真实聚合态
tease() {
  // 业务态优先：若正在 running/failed/waiting，不打断真实工作动画
  if (this.lastState &&
      ["running", "failed", "waiting"].includes(this.lastState.mode)) return;

  const saved = this.aggregate(this.currentTabs);  // 记住真实态
  if (this.teaseTimer) clearTimeout(this.teaseTimer);
  this.patrol?.stop();                              // 巡游让位
  this.applyState({ ...saved, mode: "jumping" });
  this.teaseTimer = setTimeout(() => {
    this.teaseTimer = null;
    this.push(this.aggregate(this.currentTabs));  // 恢复真实态（idle 则自动重启巡游）
  }, 2500);
}
🔒
优先级保证。

逗弄优先级高于 patrol、review，但低于 running/failed/waiting——任务真在跑时不接受逗弄打断，避免庆祝/工作动画被用户随手一点就盖掉。这与 §3.1 的优先级表一致。
